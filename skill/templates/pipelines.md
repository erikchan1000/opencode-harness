# Harness — Pipelines (parallel execution plan)

> Phase 2 artefact. Generated by `build_pipelines.py` from
> `findings.md` metadata. Tells the orchestrator which findings can
> run together and which have to wait.

## How the DAG is built

The `build_pipelines.py` script builds a directed graph:

- Add an edge `F02 → F01` if F02's `Depends on` lists F01.
- Add an edge `F02 → F01` if F02's `Touches files` overlaps with
  F01's `Touches files`. (File conflicts are dependencies even
  without a semantic relationship.)

Topologically sort into waves. A wave is a set of findings whose
dependencies all live in earlier waves.

Inside a wave, every finding can run in parallel:
- No semantic dependency on any other finding in the wave.
- No file overlap with any other finding in the wave.

Tie-breaks for putting findings earlier vs later:
1. Higher severity first.
2. More dependents first (unblocks more later waves).
3. Smaller scope first (quick green test results).

## Output format

```markdown
# Pipelines — <objective short title>

Generated by build_pipelines.py at <timestamp>.

## Summary

- Total findings: <N>
- Waves: <K>
- Max parallelism: <M> (size of widest wave)
- Critical-path length: <K> waves

## Wave 1 (parallel — <count> findings)

- **F01** — <title>
  - severity: <high|medium|...>
  - touches: `<file>`, `<file>`
- **F02** — <title>
  - severity: <high|medium|...>
  - touches: `<file>`

## Wave 2 (after wave 1, parallel — <count> findings)

- **F03** — <title>
  - depends on: F01
  - touches: `<file>`
- **F04** — <title>
  - touches: `<file>`   ⚠ shared with F02 in wave 1, deferred to here
                          to avoid file-conflict race

## Wave 3 (after wave 2 — sequential, <count> finding)

- **F05** — <title>
  - depends on: F01, F03

## Deferred / out of budget

- **F06** — <title> — sev=low, would extend critical path; backlog.
```

## Annotations

Mark findings that ended up in a later wave for **non-semantic**
reasons (file conflict, severity tie-break) with a `⚠` and a
one-line rationale. This is the audit trail; future iterations of
this harness, or a human reviewer, need to see why F04 was not in
Wave 1 even though it had no `Depends on`.

## Validation rules

The orchestrator should refuse to start Phase 3 if `pipelines.md`:

- Has any wave where two findings share a file in `Touches files`.
- Has any finding referenced as a dependency that is not in an
  earlier wave (or in the same wave — that is a cycle).
- Has any finding from `findings.md` that is neither in a wave nor
  in the deferred section.

## Re-planning during execution

If a wave exposes new findings (e.g. impl agent surfaces a related
bug), DO NOT silently add them to the current wave. Stop, escalate
to the parent agent, which decides whether to:

- Add the new finding to a future wave (re-run `build_pipelines.py`
  with the updated `findings.md`), or
- Defer it to a follow-up harness round.
